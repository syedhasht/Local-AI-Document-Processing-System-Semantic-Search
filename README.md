# Local AI Document Processing

A fully **offline**, open-source CLI tool that ingests documents, classifies them into categories, extracts structured fields, and provides semantic search — all running locally without any hosted APIs.

---

## Project Structure

```
project/
├── data/
│   ├── input/              ← Sample documents (generated by prepare_sample_dataset.py)
│   ├── embeddings.npy      ← Saved document embeddings (auto-generated)
│   └── doc_index.json      ← Document filename index (auto-generated)
├── src/
│   ├── utils.py            ← Text cleaning, file discovery, logging
│   ├── ingest.py           ← PDF / TXT / Image (OCR) ingestion
│   ├── classify.py         ← Keyword + regex classification rules
│   ├── extract.py          ← Structured field extraction per class
│   └── retrieval.py        ← SentenceTransformer embeddings + cosine similarity
├── main.py                 ← CLI entry point
├── prepare_sample_dataset.py ← Populates data/input/ from the assessment datasets
├── requirements.txt
└── output.json             ← Generated results
```

## Setup

### 1. Prerequisites
- Python 3.10 or later
- A virtual environment (recommended)

### 2. Create and activate the virtual environment
```bash
# Windows
python -m venv .venv
.venv\Scripts\activate

# macOS / Linux
python -m venv .venv
source .venv/bin/activate
```

### 3. Install dependencies
```bash
pip install -r requirements.txt
```

### 4. Prepare sample documents
```bash
python prepare_sample_dataset.py --dataset-root "C:\path\to\Assesment Task"
```
Optional flags:
```
--invoices N     Number of invoice TXT files (default: 5)
--resumes  N     Number of resume PDFs (default: 5)
--utility  N     Number of utility JPGs (default: 3)
--clear          Clear data/input/ before repopulating
```

---

## Running the Pipeline

### Process all documents
```bash
python main.py process
```
This will:
1. Read all `.pdf` and `.txt` files from `data/input/`
2. Classify each document
3. Extract structured fields
4. Build the semantic search index
5. Write results to `output.json`

### Process with OCR (for JPG/PNG images)
```bash
python main.py process --ocr
```

### Semantic search
```bash
python main.py search "Find all documents mentioning payments due in January"
python main.py search "software engineer Python experience" --top 3
```

## Bonus: Local Question-Answering with a Local LLM

`qa_local_llm.py` extends the retrieval pipeline into a full **RAG (Retrieval-Augmented Generation)** system using open-source local models.

### How it works
```
Query → Semantic Search → Top-K docs → Context → Local LLM → Answer
```

### Usage (no model required — shows retrieved context)
```bash
python qa_local_llm.py "Which invoices mention January payments?"
```

### With a local GGUF model (llama-cpp, CPU-friendly)
```bash
pip install llama-cpp-python
# Download a model, e.g. Mistral 7B Instruct Q4:
# https://huggingface.co/TheBloke/Mistral-7B-Instruct-v0.1-GGUF

python qa_local_llm.py "Summarize all utility bills" \
    --model-path "C:\models\mistral-7b-instruct.Q4_K_M.gguf" \
    --backend llama-cpp
```

### With HuggingFace transformers (TinyLlama ~600MB)
```bash
pip install transformers accelerate

python qa_local_llm.py "Who has the most work experience?" \
    --model-path "TinyLlama/TinyLlama-1.1B-Chat-v1.0" \
    --backend transformers
```

> **Design choice**: The main project does NOT require any LLM download. `qa_local_llm.py` is a standalone bonus that gracefully degrades — showing retrieved context with setup instructions if no model is provided.

## Model Downloads & Offline Usage

> **⚠️ First-time setup requires a one-time internet connection.**

Two model families are downloaded automatically on first run:

| Model | Purpose | Size | Cache location |
|-------|---------|------|----------------|
| `sentence-transformers/all-MiniLM-L6-v2` | Semantic search embeddings | ~90 MB | `~/.cache/huggingface/` |
| `easyocr` English model (only if `--ocr` used) | Image OCR | ~200 MB | `~/.EasyOCR/` |

**After the first run, everything is fully offline.** No network calls are made during processing or search.

### Pre-downloading models for strict offline environments
```bash
# Pre-download the embedding model
python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('sentence-transformers/all-MiniLM-L6-v2')"

# Pre-download easyocr (only if using --ocr)
python -c "import easyocr; easyocr.Reader(['en'])"
```

## How Classification Works

Classification is **purely rule-based** — no ML model is needed for this step.

Each document receives a score for each class based on keyword matches in its text:

| Class | Key trigger words |
|-------|-----------------|
| `Invoice` | invoice, bill to, qty, stock_code, total amount, payment due |
| `Resume` | resume, curriculum vitae, experience, education, skills, objective |
| `Utility Bill` | electricity, kwh, meter reading, units consumed, account number |
| `Other` | bank statement, salary slip, itr, form 16, cheque, check no |
| `Unclassifiable` | No class scores ≥ 2 keyword hits |

The class with the **highest score** wins. Filenames also provide a fast-path signal (e.g. `invoice_001.txt` → Invoice).

**Why rule-based?** It is transparent, deterministic, and easy to audit. For an assessment setting, this is more explainable than a black-box classifier.

## How Field Extraction Works

Extraction uses **regex patterns** tailored to each document class:

### Invoice
| Field | Pattern strategy |
|-------|----------------|
| `invoice_number` | `invoice no/number/# : <value>` |
| `date` | ISO, slash/dash formats, "15 Jan 2024" |
| `company` | "From:", "Sold by:", or first non-header line |
| `total_amount` | "Total Amount:", "Grand Total:", "Total:" |

### Resume
| Field | Pattern strategy |
|-------|----------------|
| `name` | First short line (≤6 words) not containing email or long numbers |
| `email` | Standard email regex |
| `phone` | Flexible: +1-555-1234, (555) 123-4567, +91 9876543210 |
| `experience_years` | "X years of experience", "X+ yrs experience" |

### Utility Bill
| Field | Pattern strategy |
|-------|----------------|
| `account_number` | "Account No/Number: <value>" |
| `date` | Same multi-format date regex |
| `usage_kwh` | "123.5 kwh", "456 units" |
| `amount_due` | "Amount Due:", "Total Due:", "Payable:" |

Fields not found are omitted from `output.json` (not set to null).

## How Semantic Search Works

1. **Indexing (at process time)**:
   - Each document's text (truncated to 512 chars) is embedded using `all-MiniLM-L6-v2`.
   - Embeddings (L2-normalized) are saved to `data/embeddings.npy`.
   - Filenames are saved to `data/doc_index.json`.

2. **Searching**:
   - The query string is embedded with the same model.
   - **Cosine similarity** is computed between the query vector and all document vectors using `sklearn.metrics.pairwise.cosine_similarity`.
   - Top-k results are returned ranked by similarity score.

This approach is local, fast for small-to-medium document sets, and requires no external vector database.

## Output Format

`output.json` is written after every `python main.py process` run:

```json
{
  "invoice_001.txt": {
    "class": "Invoice",
    "invoice_number": "INV-1001",
    "date": "2024-01-15",
    "company": "Acme Corp",
    "total_amount": "1250.00"
  },
  "resume_accountant_12345678.pdf": {
    "class": "Resume",
    "name": "John Smith",
    "email": "john@example.com",
    "phone": "+1-555-1234",
    "experience_years": 5
  },
  "utility_bill_01.jpg": {
    "class": "Utility Bill"
  }
}
```

> **Note**: Image files without `--ocr` will be classified as `Unclassifiable` since no text can be extracted.

## Offline / Local-Only Limitations

| Limitation | Impact |
|-----------|--------|
| Keyword-based classification | May misclassify documents with unusual formatting |
| Regex extraction | Structured fields may not be found in free-form or scanned documents |
| OCR quality | easyocr accuracy varies with image quality; skewed/low-res scans may fail |
| Embedding model size | `all-MiniLM-L6-v2` is fast but smaller models have lower semantic accuracy |
| No LLM extraction | For complex extraction (e.g. multi-page invoices), an LLM would be more robust |
| FAISS not used | sklearn cosine similarity is sufficient for ≤10,000 docs; swap to FAISS for large scale |
